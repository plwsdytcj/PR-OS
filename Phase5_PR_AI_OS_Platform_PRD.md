## 1. 第五期一句话

**第五期将前四期沉淀的多源 KOL 数据、媒介经验、甲方偏好、博主商业档案和合作响应数据整合成 PR 行业的 AI OS，形成从品牌 brief 理解、传播策略生成、KOL 匹配、需求分发、舆论推演、方案确认到投后案例回流的完整闭环。**

## 2. 阶段定位

### 2.1 前四期完成了什么

```text

```

> **把这些模块从“工作流工具”升级成“PR 行业智能决策与任务分发系统”。**

## 3. 平台定位

PR AI OS 是 PR 行业的 AI 操作系统层。

它不是：

- 单一达人库；
- 单一投放平台；
- 单一数据 API；
- 单一方案生成工具；
- 单一媒介 CRM。

它是：

```text
PR 数据层
      +
媒介决策层
      +
甲方协作层
      +
博主商业资产层
      +
传播任务分发层
      +
舆论推演与复盘层
```

核心价值：

> **把 PR 行业中原本靠人脉、经验、Excel、微信群、PPT 和主观判断完成的传播工作，转化成 AI 可理解、可匹配、可推演、可分发、可复盘的系统。**

---

## 4. 核心目标

第五期要证明：

1. 品牌 brief 可以被系统稳定理解；
2. 系统可以基于品牌目标生成传播策略；
3. 系统可以从多源 KOL 数据中匹配可执行达人组合；
4. 系统可以结合真实响应数据判断可执行性；
5. 系统可以做投放前舆论风险推演；
6. 系统可以生成多方案对比；
7. 投后数据和案例可以回流到 KOL Profile；
8. 甲方偏好、博主响应和投后结果可以持续优化推荐；
9. 平台可以支撑品牌、媒介、博主三方协作。


#### 7.6.1 MiroFish Adapter

MiroFish 可以作为 Campaign Stress Test 的可插拔推演引擎。

它不作为主系统，也不作为达人库，不负责 KOL 数据接入、报价判断、预算分配或交易撮合。

MiroFish 的定位是：

> **把投放方案作为 seed，构建多智能体传播模拟环境，帮助识别潜在传播路径、评论区风险、人群反应差异和舆论压力点。**

MiroFish 不对外承诺：

- 不承诺预测爆款；
- 不承诺预测真实 ROI；
- 不承诺替代投后归因；
- 不承诺自动判断最终投放成败。

MiroFish 对外表达应该是：

```text
投放前压力测试
舆论风险模拟
传播路径推演
多角色反馈模拟
```

不应该表达为：

```text
爆款预测
ROI 预测
真实传播结果预测
```

输入：

```json
{
  "campaign_profile": {
    "industry": "汽车",
    "product": "新能源 SUV",
    "budget": 500000,
    "goals": ["新品预热", "品牌背书", "曝光"],
    "target_audience": ["25-40岁男性", "一二线城市", "科技兴趣人群"]
  },
  "campaign_plan": {
    "plan_name": "方案B：破圈型",
    "creator_roles": ["AIGC视觉达人", "汽车垂类测评", "小红书生活方式"],
    "content_directions": ["科技感视觉大片", "真实试驾体验", "城市通勤场景种草"],
    "budget_allocation": {
      "douyin": 220000,
      "xiaohongshu": 150000,
      "dongchedi": 90000,
      "reserve": 40000
    }
  },
  "risk_context": {
    "known_risks": ["AIGC真实性争议", "价格评论风险"],
    "brand_sensitivity": "high"
  }
}
```

输出：

```json
{
  "simulation_summary": "该方案具备较强声量潜力，但 AIGC 内容真实性争议和价格评论风险较高。",
  "positive_reactions": [
    "科技感视觉内容容易吸引年轻男性用户讨论",
    "汽车垂类测评有助于建立专业背书"
  ],
  "negative_reactions": [
    "部分用户可能质疑价格偏高",
    "AIGC 内容可能被认为缺少真实使用体验"
  ],
  "risk_points": [
    "评论区价格争议",
    "AIGC 内容真实性争议",
    "达人广告密度过高导致信任下降"
  ],
  "optimization_suggestions": [
    "增加真实车主体验型内容",
    "在视觉内容后增加垂类达人解释",
    "保留机动预算用于表现较好内容的二次放大"
  ],
  "final_recommendation": "建议将该方案作为破圈补充，不建议作为唯一主方案。"
}
```

系统集成方式：

```text
Campaign Plan
      ↓
Stress Test Adapter
      ↓
MiroFish / LLM 多角色模拟 / 其他推演引擎
      ↓
Simulation Report
      ↓
方案风险提示和优化建议
```

第一版没有 MiroFish 时，可以先使用 LLM 多角色模拟作为本地 fallback。

## 8. User Stories

### 8.1 品牌方

#### Story B5-1：品牌自助输入 brief

作为品牌市场负责人，我希望输入产品、预算和传播目标，系统能生成传播策略和 KOL 推荐方案。

验收标准：

- 支持自然语言 brief；
- 支持结构化字段确认；
- 输出传播策略；
- 输出推荐达人；
- 输出预算和风险。

#### Story B5-2：品牌比较多套方案

作为品牌方，我希望比较稳健型、破圈型、性价比型方案，这样我可以根据风险和目标做决策。

验收标准：

- 至少输出 2-3 套方案；
- 每套方案有预算、达人组合、内容方向；
- 每套方案有优缺点；
- 每套方案有风险提示；
- 系统给出推荐主方案。

#### Story B5-3：品牌查看投放前推演

作为品牌方，我希望看到每套方案可能带来的用户反馈和舆论风险，这样我可以提前优化传播策略。

验收标准：

- 输出正向反馈；
- 输出负向反馈；
- 输出风险点；
- 输出优化建议；
- 明确说明推演不是 ROI 预测。

### 8.2 媒介公司

#### Story M5-1：媒介管理完整项目

作为媒介公司，我希望从 brief、推荐、甲方反馈、博主响应到投后案例都在一个项目里管理。

验收标准：

- 项目有完整时间线；
- 包含 brief、推荐、方案、反馈、响应、案例；
- 支持按项目复盘；
- 支持归档。

#### Story M5-2：媒介沉淀团队经验

作为媒介公司，我希望系统记录每次推荐、甲方反馈和投后表现，这样团队经验可以被复用。

验收标准：

- 记录推荐理由；
- 记录甲方选择；
- 记录博主响应；
- 记录投后反馈；
- 后续推荐可引用历史经验。

### 8.3 博主侧

#### Story K5-1：博主获得更匹配 brief

作为博主，我希望收到更符合我内容能力、报价和合作偏好的品牌需求，这样我可以减少无效沟通。

验收标准：

- brief 与博主标签匹配；
- 支持设置不接行业；
- 支持设置报价区间；
- 支持记录不接原因；
- 后续推送能避开不匹配需求。

#### Story K5-2：博主沉淀投后案例

作为博主，我希望把完成的合作沉淀成案例，这样未来能提升我的商业价值。

验收标准：

- 合作完成后可生成案例；
- 可填写效果数据；
- 可设置可见性；
- AI 更新商业画像；
- 后续推荐可引用该案例。

## 11. 权限与治理

### 11.1 数据隔离

- 媒介公司私有数据默认不共享；
- 甲方项目数据默认归项目空间；
- 博主公开信息和私密商务信息分层；
- 案例可见性需明确授权；
- 内部备注、底价、压价空间默认仅媒介可见。

### 11.2 角色权限

| 角色 | 权限 |
|---|---|
| 平台管理员 | 管理系统、数据源、全局配置 |
| 媒介管理员 | 管理团队、客户、达人库、项目 |
| 媒介成员 | 创建方案、推送 brief、管理反馈 |
| 甲方管理员 | 创建/查看项目、确认方案 |
| 甲方成员 | 查看方案、评论、确认候选 |
| 博主 | 维护档案、响应 brief、上传案例 |
| MCN 管理员 | 管理旗下达人、响应需求 |

### 11.3 合规原则

- 明确数据来源；
- 标记数据更新时间；
- 标记数据可信度；
- 区分公开数据、授权数据、内部数据；
- 支持数据删除和撤回；
- 不承诺真实 ROI 预测；
- 推演结果标记为辅助决策。

## 13. MVP 范围

M5 的 MVP 不需要一次完成全部 OS 能力。

### 13.1 必须做

1. 品牌自助 brief 输入；
2. 多方案生成；
3. KOL 推荐与可执行性评分；
4. 甲方协作确认；
5. brief 推送与博主响应；
6. 投放前风险推演；
7. 投后案例录入；
8. 数据回流到 KOL Profile；
9. 项目级时间线；
10. 权限和数据隔离。

### 13.2 可选做

1. 多智能体推演；
2. 自动生成 PPT；
3. 行业榜单；
4. 竞品传播分析；
5. 评论区情绪监测；
6. 投中监测；
7. 预算模拟器；
8. MCN 组织后台。

### 13.3 暂不做

1. 不做自动支付；
2. 不做担保交易；
3. 不做合同法务流；
4. 不做真实 ROI 承诺；
5. 不做平台官方全量数据替代；
6. 不做完全无人审核投放。

## 14. 成功标准

第五期成功标准：

1. 品牌可以从 brief 到方案完成自助流程；
2. 媒介可以在系统中管理完整项目；
3. 博主可以维护档案并响应需求；
4. 系统推荐可以结合甲方偏好和博主响应数据；
5. 投放前推演能帮助识别风险；
6. 投后案例可以回流并影响后续推荐；
7. 平台形成跨项目的数据复利；
8. 用户感知到系统不只是工具，而是 PR 决策和协作底层。

---

## 15. 核心结论

第五期的产品不是“更多功能”，而是把前四期沉淀的数据和流程连成闭环。

最终 PR AI OS 的核心能力是：

```text
理解品牌传播需求
评估内容节点价值
生成传播策略
匹配可执行 KOL 组合
推演舆论风险
分发合作任务
沉淀投后案例
持续优化推荐系统
```

它的长期壁垒不是某一个 API，也不是某一个达人库，而是：

> **跨品牌、跨媒介、跨博主、跨案例沉淀下来的 PR 决策数据和 AI 判断能力。**

这就是 PR 行业的 AI OS 层。
