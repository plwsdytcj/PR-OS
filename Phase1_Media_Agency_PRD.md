# PR AI OS 第一期 PRD：媒介公司 AI 工作台

产品暂定名：**PR AI OS - Media Copilot**  
阶段：**Phase 1 / M1**  
目标用户：**PR 公司、媒介公司、KOL 投放团队、内容营销代理商**  
文档版本：v0.1  
日期：2026-06-03

---

## 1. 第一期一句话

**第一期先做给媒介公司用的 AI 工作台，帮助他们通过 API、Excel、达人链接和人工补充等多种方式接入 KOL 数据，自动生成达人商业画像，并根据甲方 brief 推荐达人组合、判断风险、分配预算、生成投放方案。**

---

## 2. 为什么第一期先做媒介公司

PR AI OS 的长期方向是 PR 行业 AI OS，但第一期不直接做开放双边平台。

第一期优先服务媒介公司，原因是：

1. 媒介公司已有真实业务场景；
2. 媒介公司每天都要筛 KOL、做推荐表、写方案；
3. 媒介公司手里已有历史合作数据、报价、案例和主观经验；
4. 甲方直接使用需要更完整的交付和服务；
5. 博主开放入驻会面临冷启动和真实性验证问题。

所以第一期产品不是“开放 KOL 平台”，而是：

> **媒介公司的 AI Creator Intelligence 工作台。**

---

## 3. 第一期核心目标

第一期要证明：

1. 多源 KOL 数据可以被接入系统；
2. 不同来源的数据可以被清洗、去重、结构化；
3. 系统可以生成统一的 KOL 商业画像；
4. 媒介输入甲方 brief 后，系统可以推荐合适的达人；
5. 系统可以输出推荐理由、报价判断、风险提示和预算建议；
6. 系统可以生成一份甲方可读的投放方案。

第一期不追求：

- 不追求直接连接所有平台官方 API；
- 不追求完整交易闭环；
- 不追求真实 ROI 预测；
- 不追求自动下单；
- 不追求开放博主注册。

---

## 4. 第一期产品形态

第一期产品由 5 个核心模块组成：

```text
达人数据连接层
      ↓
达人数据清洗与统一画像
      ↓
AI 商业标签与价值判断
      ↓
甲方 brief 匹配推荐
      ↓
方案生成与导出
```

更具体的工作流：

```text
媒介公司通过 API / Excel / 链接 / 人工录入接入达人数据
        ↓
系统清洗、去重、字段映射
        ↓
生成统一 KOL Profile
        ↓
AI 分析内容、案例、报价、风险
        ↓
媒介输入甲方 brief
        ↓
系统推荐达人组合
        ↓
输出推荐理由、预算分配、风险提示
        ↓
生成 Markdown / Excel 投放方案
```

## 5. 核心模块一：达人数据连接层

### 5.1 模块定位

达人数据连接层是第一期的底座。

它不假设所有数据都来自 Excel，也不假设所有数据都能通过一个 API 获得。

它要支持：

- 第三方 KOL 数据 API；
- 媒介公司内部 Excel / CSV；
- 达人主页链接；
- 内容链接；
- 历史合作案例；
- 人工补充信息。

统一目标是：

> **把各种来源的 KOL 信息统一成平台自己的标准达人数据模型。**

### 5.2 支持的数据入口

#### 入口 1：API Connector

第一期支持可配置 API Connector。

候选数据源：

- OneAPI / GetOneAPI；
- 紫薯通告；
- KOLRank；
- CloudKOL；
- 其他第三方 KOL 数据服务。

第一期优先级：

1. OneAPI：用于技术验证，拉取公开账号、作品、互动数据；
2. 紫薯通告：用于验证媒介商业字段，例如报价、联系方式、蒲公英/星图相关数据；
3. 其他 API：作为后续扩展。

API Connector 需要支持：

- 配置 API Key；
- 配置平台；
- 输入达人 ID / 主页链接 / 内容链接；
- 拉取账号基础信息；
- 拉取作品列表；
- 拉取单条作品数据；
- 拉取互动数据；
- 记录 API 来源和更新时间；
- API 失败时允许人工补充。

#### 入口 2：Excel / CSV 导入

媒介公司可以拖拽上传已有数据。

支持文件类型：

- `.xlsx`
- `.csv`

典型文件：

- 达人资源表；
- 刊例表；
- 报价表；
- 历史投放表；
- 合作案例表；
- 项目复盘表。

导入能力：

- 自动识别表头；
- 支持字段映射；
- 支持多 sheet；
- 支持预览数据；
- 支持去重；
- 支持错误行提示；
- 支持导入后生成 KOL Profile。

#### 入口 3：达人主页链接解析

媒介可以输入或批量粘贴达人主页链接。

支持平台：

- 抖音；
- 小红书；
- B 站；
- 快手；
- 微博；
- 视频号，第一期可先手动补充；
- 公众号，第一期可先手动补充。

系统处理：

- 识别平台；
- 提取达人 ID；
- 调用对应 API 或解析公开字段；
- 生成基础账号信息；
- 关联已有达人，避免重复建档。

#### 入口 4：内容链接解析

媒介可以输入单条作品链接。

系统处理：

- 识别平台；
- 提取内容标题、文案、互动数据；
- 识别内容主题；
- 识别品牌提及；
- 判断是否疑似商业内容；
- 归档到对应达人；
- 作为 AI 打标签依据。

#### 入口 5：人工补充

因为很多商业信息 API 拿不到，第一期必须支持人工补充。

人工字段包括：

- 报价；
- 联系方式；
- 档期；
- 是否好沟通；
- 是否配合改稿；
- 是否按时交付；
- 历史合作品牌；
- 甲方反馈；
- 压价空间；
- 风险备注；
- 媒介主观评级。

这些信息是媒介公司的核心私有资产，也是 PR AI OS 的长期壁垒。

### 5.3 数据源优先级

当多个数据源冲突时，优先级建议：

```text
人工确认数据 > 内部历史合作数据 > 授权/API 商业数据 > 公开 API 数据 > 自动解析数据
```

例如：

- 报价以媒介手动确认或内部历史报价为准；
- 粉丝量以最新 API 数据为准；
- 是否好沟通以媒介人工评价为准；
- 内容领域可以由 AI 生成，但允许人工覆盖。

## 8. 甲方 Brief 匹配推荐

### 8.1 Brief 输入

媒介输入甲方需求：

```text
我们是新能源汽车品牌，预算 50 万，准备做新能源 SUV 新品上市预热。
目标用户是 25-40 岁一二线城市男性，希望突出科技感、智能化和高端感。
平台优先抖音、小红书、懂车帝。
```

系统解析为：

```json
{
  "industry": "汽车",
  "product": "新能源 SUV",
  "budget": 500000,
  "campaign_stage": "新品上市预热",
  "goals": ["曝光", "品牌背书", "新品预热"],
  "target_audience": ["25-40岁男性", "一二线城市", "科技兴趣人群"],
  "platform_preference": ["抖音", "小红书", "懂车帝"],
  "content_preference": ["科技感", "智能化", "高端感"]
}
```

### 8.2 匹配评分

| 维度 | 权重 |
|---|---:|
| 行业匹配度 | 20% |
| 内容能力匹配度 | 15% |
| 传播阶段匹配度 | 15% |
| 人群匹配度 | 15% |
| 案例背书 | 10% |
| 报价合理性 | 10% |
| 风险控制 | 10% |
| 数据可信度 | 5% |

### 8.3 推荐结果

每个推荐达人输出：

- 匹配分；
- 推荐等级；
- 推荐角色；
- 建议内容形式；
- 建议预算；
- 推荐理由；
- 报价判断；
- 风险提示；
- 数据来源；
- 是否需要人工核验。

## 11. 第一期 API 策略

第一期不把 API 当作核心壁垒，而是把 API 当作数据源。

推荐策略：

```text
主 PoC API：OneAPI
商业字段验证：紫薯通告
后续扩展：KOLRank / CloudKOL / 飞瓜 / 千瓜 / 新红
```

### 11.1 OneAPI 用途

用于验证：

- 通过达人 ID 获取账号信息；
- 通过主页链接获取达人信息；
- 获取作品列表；
- 获取作品详情；
- 获取互动数据；
- 获取评论数据，视接口支持情况而定。

### 11.2 紫薯通告用途

用于验证：

- 商业报价；
- 联系方式；
- 蒲公英/星图相关字段；
- 媒介投放场景数据；
- 企业 API / webhook 可用性。

### 11.3 接 API 前必须确认的问题

1. 支持哪些平台？
2. 是否支持通过达人 ID / 主页链接查询？
3. 返回字段有哪些？
4. 是否有粉丝数、作品列表、互动数、评论？
5. 是否有报价、联系方式、蒲公英/星图字段？
6. 数据来源是否合法？
7. 是否允许商用？
8. 是否能签企业合同和 SLA？
9. 单次调用价格是多少？
10. 是否限流？
11. 失败率如何？
12. 数据更新频率是多少？

## 12. 成功标准

第一期成功标准：

1. 媒介可以通过 Excel、链接、API、人工录入至少 3 种方式接入达人数据；
2. 系统可以把不同来源数据合并成统一 KOL Profile；
3. 系统可以识别重复达人；
4. 系统可以生成 AI 商业画像；
5. 媒介可以输入甲方 brief；
6. 系统可以推荐达人组合；
7. 推荐结果能解释为什么推荐；
8. 系统能提示报价、风险和数据可信度；
9. 系统能生成一份可发给甲方的方案；
10. 媒介用户认为它节省了筛人和写方案时间。

## 14. 核心结论

第一期不要把产品做成“上传达人表工具”，也不要做成“单一 API 查询工具”。

第一期真正要做的是：

> **一个支持多源数据接入的媒介公司 AI 工作台。**

数据来源可以是：

```text
API
Excel
CSV
达人主页链接
内容链接
历史合作案例
人工备注
```

系统价值是：

```text
把多源 KOL 数据
变成统一 KOL Profile
再变成 AI 商业画像
最终变成甲方 brief 下的推荐方案
```

这才是 PR AI OS 的第一块地基。
